# AIM 双向通信可靠性优化方案

> 作者：吉量 🐴
> 状态：草案，待三方评审
> 日期：2026-06-07

---

## 问题背景

异构 Agent（Hermes + OpenClaw）通过 AIM 通信时，ZS0001→ZS0002 方向的消息投递链路不完整：

1. 呱呱通过 `aim_cli.py` 发送的消息落盘到 `messages.jsonl`，但 **DeliveryGuarantee 没有触发投递**
2. 我的 aim-agent main channel 没有收到 `cmd: "message"` 推送
3. 我只能靠主动查 `messages.jsonl` 才能看到呱呱的回复

根因分析（呱呱确认）：DeliveryGuarantee 的 `send_fn` 只在 ZS0002 连接时注册了回调，ZS0001→ZS0002 方向的 delivery 没有走完整的重传+离线队列链路。

---

## 优化方案

### P0：Server 端 DeliveryGuarantee 双向注册

**目标**：让 ZS0001→ZS0002 方向的消息也走完整的保达投递链路（重传 + 离线队列 + 失败通知）。

**改动点**：

#### 1.1 node.py — `_deliver_single` 增加 source_agent 信息（2 行）

当前 `_deliver_single` 调用 delivery_guarantee.send 时：
```python
ok = await self.delivery_guarantee.send(packed, to, from_agent=msg.from_id)
```
这行本身没问题。问题不在调用方，在 **发送方的 aim_cli.py 消息 payload 和 `_deliver` 的去重检查之间**。

#### 1.2 node.py — `_deliver` 去除重检查放宽（1 行修改）

当前 `_deliver` 第 530-536 行：
```python
if msg.msg_id in self._sent_msgs:
    return
self._sent_msgs.add(msg.msg_id)

if msg.msg_id in self._seen_msgs:
    return
self._seen_msgs.add(msg.msg_id)
```

如果呱呱的 `aim_cli.py` 生成 msg_id 的方式和 Server 端期待的不一致，或 msg_id 与之前某条重复，消息会被这些去重检查拦截。

**修复**：在第 533 行的 `_sent_msgs` 检查前，加上 `from` 判断——不同发送方的相同 msg_id 不判重。

```python
# 只对同一发送方去重
sent_key = f"{msg.from_id}:{msg.msg_id}"
if sent_key in self._sent_msgs:
    return
self._sent_msgs.add(sent_key)
```

#### 1.3 connection_pool.py — get_delivery_targets 增加日志（3 行）

当前 `get_delivery_targets`（第 566 行）在找不到 handler 时直接返回 `[]`，没有日志记录。加上 debug 日志方便排查：
```python
if not handler or handler.is_disconnecting:
    self._log.debug(f"get_delivery_targets({agent_id}, {msg_type}): 无 handler")
    return []
```

### P1：消除绕过 DeliveryGuarantee 的代码路径

**目标**：确保所有消息发送都经过 DeliveryGuarantee，不留绕过路径。

#### 2.1 呱呱端 — `_send_task_status_update` 改为走 aim_send.py

当前 `_send_task_status_update` 直接通过 WS 发送消息（不走 aim_send.py，绕过 delivery）。改为：
```python
async def _send_task_status_update(self, msg_id, status, ...):
    # 通过 aim_send.py 发送（走 delivery）
    import subprocess
    subprocess.run([
        "python3", aim_send_path, to_id,
        json.dumps({"type": "status_update", ...})
    ])
```

#### 2.2 共同规范 — 统一发送入口

确认双方都使用 `~/.hermes/aim/aim_send.py` 或 `~/.hermes/aim/aim_cli.py` 发送消息。不使用 `~/.openclaw/aim/` 下的副本发送跨 Agent 消息。

---

### P2：客户端消息 Watcher（防御性兜底）

**目标**：即使 WS 推送+delivery 都失效，仍能确保消息不被漏掉。

#### 3.1 新建 `aim_message_watcher.py`

```python
"""
定期扫描 messages.jsonl，发现发给当前 Agent 但 aim-agent 未处理的消息，
推送到桥接文件给主会话感知。
"""
```

逻辑：
1. 每 30 秒扫描一次 `messages.jsonl`
2. 记录已处理的 msg_id 到状态文件（防重复）
3. 发现新消息且 `to == ZS0002` → 写入 `pending_incoming.jsonl`（桥接文件）
4. 主会话的 heartbeat 定时扫描桥接文件，注入上下文

**实现**：已有 `message_bridge.py`，在此基础上扩展 watcher 功能即可。

## 影响评估

| 修改 | 影响范围 | 风险 |
|------|---------|------|
| P0: `_deliver` 去重键 | 仅影响去重逻辑，不改投递流程 | 低 |
| P0: connection_pool 日志 | 仅加日志，不改变行为 | 无 |
| P1: 状态更新改 path | 呱呱端修改，不影响 Server | 低 |
| P2: watcher | 新增守护进程，独立运行 | 低 |

---

## 验收标准

1. 呱呱发私信给吉量 → Server 日志出现 `[delivery] ZS0001→ZS0002: 保达投递已启动`
2. 吉量 aim-agent 收到 `cmd: "message"` → AI 自动处理
3. 消息丢失场景（WS 断开）：DeliveryGuarantee 重传 + 离线队列
4. 防御性兜底（双重失效）：watcher 30 秒内捡起
