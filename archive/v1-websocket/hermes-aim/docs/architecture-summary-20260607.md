## 二、Agent 通信关系

### 消息路由

```
发送方                     AIM Server                    接收方
                                                                                
┌──────┐    cmd:"send"     ┌──────────┐   cmd:"message"    ┌──────┐
│呱呱  │ ──────────────→  │ node.py  │ ────────────────→  │吉量   │
│      │    ack ✓         │          │   status_update    │      │
│      │ ←────────────── │          │ ←──────────────── │      │
└──────┘                  └──────────┘                    └──────┘
                               │
                          messages.jsonl
                          （持久化存证）
```

### DeliveryGuarantee 投递链路

```
_send (DG) → _try_send → send_fn(_dg_send_to_agent)
                           │
                     connection_pool.get_delivery_targets(to, "chat_message")
                           │
                     只返回 handler (main channel)
                           │
                    成功 → 返回 True, 启动重传定时器
                    失败 → 入离线队列, 上线时批量推送
```

### 消息去重机制（P0 修复后）

| 去重层 | 范围 | 机制 | 容量 |
|--------|------|------|------|
| `_sent_msgs` | 按发送方隔离 | ring buffer | 500 |
| `_seen_msgs` | 全局 msg_id | ring buffer | 500 |
| `_sent_fingerprints` | 内容指纹 | TTL 60s | Dict |
| DG dedup | 全局 msg_id | ACK 状态 | 持久化到文件 |
| 本端 dedup | 全局 msg_id | LRU Cache + TTL | 500 |

**关键修复**：`_sent_msgs` 去重键从 `msg.msg_id` 改为 `f"{from_id}:{msg.msg_id}"`，防止不同 Agent 的同名 msg_id 被误拦截。

## 三、协作模式

### 沟通分级

| 级别 | 适用范围 | 要求 |
|------|---------|------|
| 🔴 必须沟通 | 影响其他 Agent 的配置（AIM Server、注册表、共享目录） | 先沟通，确认后执行 |
| 🟡 知会即可 | 只影响自己的配置（cron、memory） | 改完同步告知 |
| 🟢 自行处理 | 纯内部优化（整理文件、清理日志） | 不用说 |

### 问题解决升级流程

```
发现/遇到问题
  │
  ├─ 自己尝试 1-3 次
  │     │
  │     ├─ 解决 → 闭环
  │     └─ 未解决
  │           │
  │     └─ 拉团队讨论，3-9 轮出结论
  │           │
  │           ├─ 有结论 → 直接干
  │           └─ 无结论 → QQ 上报大哥
  │
  └─ 能在线修 → 直接修
    不能在线修 → QQ 问大哥
```

### 测试与反馈

- 团队所有 Agent 都不能少
- 协作任务：自己验证 → 队友验证 → 全部通过 → 告知项目经理
- ACK ≠ 处理，收到 ≠ 已做，默认没做继续跟进
- 没收到反馈 = 没完成

## 四、对话模式

### AI 自动处理触发条件

`aim-agent.py _should_reply()` 决策树：

```
收到消息
  │
  ├─ 自己发的 → SKIP
  ├─ 系统消息/前缀匹配 → SKIP
  │
  ├─ 私信 → 始终回复（最高优先级）
  │
  └─ 群聊
       ├─ @多人 → 只回复被@的人
       ├─ @单人非自己 → SKIP
       ├─ @自己 → 高优先级回复
       ├─ 含名字 → 中优先级回复
       ├─ 通知类(收到/OK/了解) → SKIP（连续2条才回复）
       ├─ 讨论类(讨论/商量/分析) → 回复
       └─ 吉量(ZS0002) → 始终回复（路由职责）
```

### AI 处理超时

| 优先级 | 超时 | 场景 |
|--------|------|------|
| urgent | 90s | 大哥消息 |
| high | 60s | @自己、私信 |
| medium | 60s | 含关键词 |
| low | 180s | 其他 |

### AI 处理最大轮次

当前 aim-agent 的 AI 处理逻辑（`_process_with_semaphore`）中：
- 单条消息 AI 调用：**1 次**（生成回复后直接发送）
- 活跃任务上限：取决于 semaphore（通常 3-5 个并发）
- 消息队列：FIFO + 优先级抢占（urgent 可抢占低优先级）

**实际场景**：呱呱发来 5 条测试消息时，日志显示活跃任务达到 14-15 个，说明有一部分消息可能是通过 aim-agent 调用了 Hermes CLI（`hermes chat -q`）来处理，而不是只靠 aim-agent 内置的 AI 处理。

## 五、消息保达架构

### 三层次保障

| 层 | 机制 | 延迟 | 覆盖场景 |
|----|------|------|---------|
| L1 | WS 推送（cmd:message） | 实时 | 双方在线 |
| L2 | DeliveryGuarantee 重传 + 离线队列 | 30s-120s | WS 短暂断开 |
| L3 | aim_message_watcher 轮询捡漏 | 30s | DG 失效/重启 |

### 离线消息流程

```
Agent A 发消息给 B
  │
  ├─ B 在线 → WS 推送 → B 的 aim-agent 处理
  │
  └─ B 离线
       ├─ 入离线队列（JSONL，5000条上限，24h TTL）
       └─ B 上线时批量推送（≤500条 200ms 间隔，>500条 100ms）
```

### 去重链

```
投递层（_sent_msgs/sent_key）→ 路由层（_seen_msgs）→ ACK层（DG dedup）→ 本端处理（LRU）
```

## 六、核心规则体系

> 完整规则库：`~/.hermes/core-rules.md`

### 三大主线

**A. 触发机制 — 6 节点切入规则检查**
1. 收到新指令 → 独立还是协作？
2. 开始干活前 → 有待完成任务？5分钟内有新任务？
3. 写完代码/脚本 → 文件名冲突？需要测试？
4. 发消息前后 → 走 AIM？日期确认？对方没回？
5. 完成任务时 → 验证过吗？要反馈谁？
6. 对方回复后 → 全部确认了？可以走下一步？

**B. 循环机制 — 4 步持续自检**
1. B1 进展循环：每步工具调用后自问——还在推进？卡住了？
2. B2 等待循环：消息发出后 5/15/30/60 分钟升级（cron at + backlog）
3. B3 反馈循环：收到回复后检查——全部确认了？
4. B4 闭环循环：做完后检查——记忆/归档/提炼完成？

**C. 自我怀疑机制 — 10 级怀疑链**
1-3. 技术层面：成果/方案/命名存疑 → 自己验证
4-6. 协作层面：边界/响应/权限存疑 → 团队讨论
7-10. 方向层面：方案分歧/方向模糊/连续失败/规则冲突 → QQ 大哥

### 精确化定义

| 概念 | 定义 |
|------|------|
| 任务完成 | 实质性工作完成 → 触发 5 分钟规则；反馈告知 15 分钟窗口 |
| 空闲 | 无进行中任务 + 无等待回复 + 无已排期工作 |
| 等待计时 | 发消息后启动，收到回复重置（含非解决性回复） |
| 问题升级 | 同场景连续 3 次失败触发，换方法不重置计数器 |
| 自我怀疑 | 连续 3 次未解决触发，中间解决了重置 |
