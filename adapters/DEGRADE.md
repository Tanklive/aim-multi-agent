# AIM Phase 2 — 降级策略

> 版本: v1.0 | 日期: 2026-06-15 | 合写: 呱呱 + 吉量

---

## 1. 降级层次

```
L1: 直接调用 adapter（最快，秒级）
  ↓ 超时/exit2/重试耗尽
L2: 文件队列（降级，分钟级）
  ↓ 30min TTL 过期
L3: JetStream 持久化（兜底，7天保留）
```

---

## 2. 触发条件

| 场景 | 触发 | 行为 |
|------|------|------|
| adapter 超时（>120s） | L1 → L2 | 写入文件队列，周期性重试 |
| exit 1 重试3次仍失败 | L1 → L2 | 写入文件队列 |
| exit 2（主动降级） | L1 → L2 | 写入文件队列 |
| exit 3（人工介入） | **不降级** | 通知大哥，停止处理 |
| 文件队列 30min TTL 过期 | L2 → L3 | JetStream 保留，人工回放 |
| Letta session 互斥 | L1 → L2 | 降级等空闲，2min 后重试 |

---

## 3. 各框架降级行为

### 3.1 Hermes（吉量）

| 降级场景 | 可能性 | 触发条件 |
|----------|--------|----------|
| hermes chat 超时 | 低 | 大哥正在对话，session 被占 |
| CLI 不可用 | 极低 | 路径变更 |
| 模型不可用 | 低 | API 配额耗尽 |

### 3.2 OpenClaw（呱呱）

| 降级场景 | 可能性 | 触发条件 |
|----------|--------|----------|
| 心跳未处理 | 高 | 主会话正在处理其他任务 |
| 队列排队 | 中 | 多个消息同时到达 |
| 降级后恢复 | 高 | 心跳周期内处理完成 |

**OpenClaw 特性**：adapter 写 `.aim-queue/` 等心跳消费，适配器级超时后走 L2 降级，但实际上消息已被写入队列，心跳会自动处理。降级到 L2 后，心跳会在下一个周期（≤5min）消费并回复。

### 3.3 Letta（小火鸡儿）

| 降级场景 | 可能性 | 触发条件 |
|----------|--------|----------|
| session 互斥 | 高 | 正在与用户对话 |
| `letta -p` 超时 | 中 | 模型响应慢 |
| CLI 不可用 | 低 | npm global 路径问题 |

## 4. 文件队列降级规范

### 4.1 写入格式

```json
{
  "id": "degrade-{timestamp}-{agent}",
  "from": "ZSxxxx",
  "to": "ZSxxxx",
  "content": "消息内容",
  "degraded": true,
  "degrade_reason": "exit2|timeout|retry_exhausted",
  "degrade_time": "ISO8601",
  "retry_count": 3
}
```

### 4.2 恢复机制

- nats-agent 后台协程每 30s 扫描降级队列
- 发现降级消息 → 重新调 adapter
- 成功 → 删除降级消息 + NATS 回复
- 仍失败 → 更新 retry_count
- 30min TTL 过期 → 从文件队列移除，消息已在 JetStream 保存

---

## 5. 大哥通知规范（exit 3）

当 adapter 返回 exit 3（需人工介入）时：

1. nats-agent 停止处理该消息
2. 通过 OpenClaw 通知大哥：
   ```
   🚨 AIM 降级告警 — {Agent} adapter 不可用
   原因: {stderr}
   消息: {content[:100]}
   操作: 需人工检查
   ```
3. 不影响其他 Agent 消息处理
4. 修复后重新加载即可恢复

---

## 6. 降级监控

| 指标 | 阈值 | 告警 |
|------|------|------|
| L1→L2 降级率 | >50% 持续10min | 通知大哥 |
| L2 队列积压 | >20条 | 通知大哥 |
| 文件队列 TTL 过期 | >10条/天 | 通知大哥 |
| exit 3 次数 | >0 | 立即通知大哥 |
