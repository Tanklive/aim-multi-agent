## 一、3轮基本测试（功能验证）

### T1. 网络断连 — 干净断开
- **操作**: Server 关闭进程（SIGTERM），然后重启
- **预期行为**:
  - Client 端 ws 连接断开 → 指数退避重连（2s→4s→...）
  - Server 重启后，Client 自动重连成功 → 自动重认证
  - 消息无法送达期间 → 离线队列缓存（最多5000条）
  - 上线后离线消息回放成功
- **恢复时间指标**: 重连 ≤ 30s（从Server重启到Client认证完成）

### T2. 网络断连 — 强制 kill
- **操作**: Server 进程 SIGKILL（无机会做优雅关闭），几秒后重启
- **预期行为**:
  - Client 端 ws ping/pong 超时后检测到断开
  - websockets ping_interval=30s, ping_timeout=10s → 最多40s检测到断连
  - 重连流程同T1
  - 离线队列不应丢消息（JSONL持久化保障）
- **恢复时间指标**: 检测+重连 ≤ 60s

### T3. Client 断连重连（正常场景）
- **操作**: 杀 aim-agent 进程 → 重启
- **预期行为**:
  - Server 在 90s 内无心跳 → heartbeat_timeout → 标记 offline
  - Client 重启后建立新连接 → 重新认证 auth_ok
  - Server 恢复状态为 online
  - 离线期间的消息上线后回放
- **恢复时间指标**: 重连 ≤ 30s

### T3.5. 消息持久化验证（呱呱建议新增）
- **操作**: Client 断连 → Server 收到消息 → 确认 messages.jsonl → Client 上线回放
- **预期行为**:
  - Client 断连后，发给该 Agent 的消息写入 `data/messages.jsonl`
  - Client 上线后，Server 读取 JSONL 并回放
  - 回放的消息与原始消息内容、顺序一致
  - 回放完成后，消息状态从 pending 转为 delivered
- **恢复时间指标**: 回放完成 ≤ 100s（500条以内）
- **来源**: 呱呱评审建议 — 双链路闭环核心保障

## 二、5轮全面测试（异常覆盖）

### T4. 心跳超时 — Client 陷入死循环/卡死
- **操作**: 暂停 Client 心跳发送
  - **首选方案**（呱呱建议）: 注释 `aim-agent.py` 心跳发送代码 → 重启 Client
  - **备选方案**: macOS pf 防火墙封端口模拟网络隔离
  - ~~SIGSTOP 方案~~（因macOS稳定性问题排除）
- **预期行为**:
  - Server 扫描（每15s）检测到该 Agent 心跳超时
  - 90s 无心跳 → AgentStateManager 标记 offline
  - 触发 lifecycle 事件 agent_offline → 清理连接池
  - 恢复后 Client 重新连接
  - 断连期间的消息回放成功
- **恢复时间指标**: 从恢复心跳到 online ≤ 15s（下一个扫描周期）

### T5. 离线队列写满
- **操作**: 大量发消息给 offline 的 Agent，触发离线队列 5000 条上限
- **注意**: 5000条上限在 **Server 端**（呱呱确认），JSONL 文件存储
- **模拟方式**: 脚本批量发送，100ms间隔，不需真实发5000条——注意测试后清理残留数据
- **预期行为**:
  - JSONL 文件达到 5000 条后静默截断，日志记录丢弃数量
  - 不影响 Server 正常运行
- **恢复时间指标**: 截断不阻塞

#### T5.1 离线队列文件不存在（吉量建议新增）
- **操作**: 删除或重命名 messages.jsonl 文件，然后触发离线消息存储
- **预期行为**:
  - Server 自动创建新的 messages.jsonl 文件
  - 不崩溃，日志记录文件缺失警告
  - 新消息正常写入新文件
- **恢复时间指标**: 文件重建 ≤ 1s

### T6. 多 Agent 同时断连后同时重连
- **操作**: 断开 ZS0001 + ZS0002 的连接，同时重连
- **预期行为**:
  - 各自独立认证：互不影响
  - 认证限流按 agent_id 隔离：不会因一个 Agent 重连导致另一个被限流
  - 各自恢复在线状态
  - 消息正确路由到对应 Agent
- **恢复时间指标**: 各自独立 ≤ 30s

### T7. Client 认证失败 — token 过期/密钥不匹配
- **操作**: 修改 ZS0002 的 token/secret，然后重连
- **预期行为**:
  - Server 返回 auth_fail + 原因说明
  - Client 重试策略（最多5次）
  - 5次后放弃连接（retry_delay 增至 30s）
  - 定期（30s）重试认证
- **恢复时间指标**: 密钥恢复后 ≤ 30s 重连成功

### T8. Websocket 连接池满 — channel 上限触发
- **操作**: 同一 Agent 创建 6 个 main channel 连接
- **构造方式**: 使用 WebSocket 客户端工具（websocat）或 Python 脚本批量建连
  - 不可依赖 aim-agent.py（一个进程只建一个连接）
- **预期行为**:
  - connection_pool 5/channel 上限触发
  - 第6个连接被拒绝（max_connections_per_channel）
  - 同 channel 只踢最旧连接（不误踢其他 channel）
- **恢复时间指标**: N/A（设计上拒绝超额连接）

### T9. 极端负载 — 高频消息 100条/30s
- **操作**: 30秒内向一个 Agent 发送 100 条消息
- **发送方式**: 使用 WebSocket 直连（Python websockets 库），不走 CLI fork
  - CLI 每次 fork 进程开销大，会成瓶颈
- **预期行为**:
  - Server 无崩溃
  - 无内存泄漏（监控 RSS）
  - status_feedback 频率限制生效（3条/s/agent）
  - 消息去重按发送方隔离（_sent_msgs 用 from_id:msg_id 做 key）
  - 所有消息最终落盘到 messages.jsonl
- **恢复时间指标**: 压力停止后资源释放 ≤ 10s

#### T9.1 超大消息（吉量建议新增）
- **操作**: 发送单条超大消息（如 1MB+ 的 payload）
- **预期行为**:
  - Server 能处理大消息，不崩溃
  - 消息正常投递或返回 size_limit 错误
  - 不影响其他消息的处理
- **恢复时间指标**: 大消息处理 ≤ 5s

### T10. 注册制 — 5条准入标准验证（与 P3-3 合并）
- **操作**（与 P3-3 合并）: 并发注册多个新 Agent
- **当前设计**:
  1. operator_valid → 操作人存在且 active
  2. agent_limit → 未超出操作人上限
  3. info_valid → 基本信息完整
  4. no_duplicate → 无冲突注册
  5. rate_limit → 限流保护
- **恢复时间指标**: 注册请求 3s 内响应

#### T10.1 注册冲突（吉量建议新增）
- **操作**: 并发注册同一 agent_id（如两个客户端同时注册 ZS0099）
- **预期行为**:
  - 第一个注册成功，第二个返回 conflict 错误
  - Server 不崩溃，状态一致
  - 已注册的 Agent 不受影响
- **恢复时间指标**: 冲突检测 ≤ 1s

## 三、P2 测试项（呱呱建议新增）

### T11. 连接池耗尽排队/拒绝（呱呱建议）
> 同一 Agent 已达 20 连接上限时新连接的行为
- **前置条件**: 让 ZS0001 建立 20 个 main channel 连接（模拟脚本批量创建 WebSocket 连接）
- **操作**:
  1. 使用 Python websockets 库模拟 20 个连接（用不同 WS key/UA）
  2. 再尝试建立第 21 个 main channel 连接
- **预期行为**:
  - 第 21 个连接被拒绝（max_connections_per_channel 或连接池上限触发）
  - 返回明确的拒绝原因（连接池满提示）
  - Server 自身不受影响，已有 20 个连接正常运行
- **恢复时间指标**: N/A（设计上拒绝超额连接）

### T12. 离线队列磁盘满降级（呱呱建议）
> JSONL 写入失败时的降级行为
- **操作**: 模拟磁盘满状态（填满 /tmp 或限制 AIM Server 数据目录的磁盘配额）
- **模拟方式**:
  - 使用 dd 或 fallocate 填满数据目录所在分区（注意安全，建议用专门的 tmpfs）
  - 或通过 Python 脚本直接 mock open().write() 返回 OSError
- **预期行为**:
  - Server 写入 messages.jsonl 失败时捕获异常
  - 日志记录写入失败（含错误详情）
  - Server 自身不崩溃
  - 恢复磁盘空间后自动恢复写入
- **恢复时间指标**: 磁盘恢复后 ≤ 5s 恢复正常写入
