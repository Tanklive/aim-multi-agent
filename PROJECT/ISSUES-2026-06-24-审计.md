## 八、隐患清单（系统级风险）

### R-001: 通知消费侧断裂 🔴

**场景**：火鸡儿 21:19 @呱呱 提建议 → NOTIFICATION.md 写入 mentioned.jsonl ✅ → 但无 daemon 消费推送 → 呱呱 21:23 才通过 AIM inbox 发现。
**根因**：notification.py 的 `_emit_system_event()` NATS 通道已就绪，但三方都未订阅自己的 `aim.notification.<id>`。
**影响**：@提醒无法实时触达，依赖心跳轮询 inbox（最长 30s 延迟）。
**建议**：三方各在自己 runtime 加订阅 `aim.notification.<agent_id>`，呱呱侧通过 cron polling 兜底。

### R-002: StallWatchdog 停摆 🔴

**场景**：StallWatchdog 自 6/21 16:42 停止，日志文件完全消失。今天是 6/24，已停 3 天。
**根因**：未知。可能是进程崩溃后 launchd 未重启（plist 缺 SuccessfulExit=false？）
**影响**：三 Agent 无自愈能力。任何 dispatch 死锁/超时将永久卡住，无人恢复。
**建议**：立即排查 plist + 重启 StallWatchdog。

### R-003: ZS0002 adapter 版本分裂 + context-card token 爆炸 🔴

**场景**：ZS0002 shared adapter v2.0（API Server mode, 64行）vs deploy v1.3（CLI mode, 146行）。deploy 版注入 context-card 导致缓存命中 99%→23%，每次推理 ~27K tokens。
**影响**：API Server 优化效果被 context-card 膨胀抵消。每条消息多耗 ~5K tokens。
**建议**：① 同步 deploy→v2.0 ② context-card 按消息类型注入（非 TASK 只注角色卡）。

### R-004: 三端 adapter 全版本不一致 🟡

**现象**：sync-check.sh 报告 ZS0002 不一致。实际三端都不同步。
**根因**：shared/adapters/ 是参考副本但无人维护。修改在本地 deploy 目录，shared 没更新。
**影响**：恢复/迁移时找不到运行版本。新 Agent 接入无参考。
**建议**：① 三方统一「改完自己 adapter → 同步 shared」流程 ② sync-check.sh 扩展覆盖三端。

### R-005: _processed_ids 持久化容量 🟡

**场景**：6/24 加了 JSONL 持久化，但无容量上限。4465 行 main.py 每条消息处理写一次。
**风险**：1 个月后 JSONL 可能达到数 MB，启动恢复时间线性增长。
**建议**：加 2000 条 LRU 上限，类似 observer.jsonl 的 rotate 机制。

### R-006: JetStream 旧配置残留 🟢

**现象**：consumer retention=max_age 配置可能未更新。nsc 重建 consumer 从未执行。
**影响**：旧消息可能在 JetStream 永久保留，消耗磁盘。
**建议**：吉量执行 nsc consumer 重建。

### R-007: 无跨 Agent 统一部署工具 🟡

**场景**：deploy-verify.sh 只检查文件 MD5，不做功能测试。deploy_sdk.sh 只覆盖 SDK。
每次部署靠人工发消息验证。
**影响**：NOTICE 1.3.0「发版后无人实测→合写 bug」可能重现。
**建议**：Phase 2 优先加端到端 smoke test（参考 AIM Client 定位：发送→接收→验证回复）。
