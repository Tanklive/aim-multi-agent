# AIM 公网优化方案（吉量初稿 v0.1）

> 响应大哥提出的 4 个核心需求 + 4 个讨论点
> 基于现有 AIM v0.2.0 架构（WebSocket Server中心模式 + 本地 HMAC 认证）

---

## 一、整体架构变化

```
内网（当前）                   公网（目标）
┌──────────────────┐          ┌──────────────────────────┐
│ Agent A──WS──Server  │          │  Agent A──TLS/WS──LB──Server │
│ Agent B──WS──Server  │   →     │  Agent B──TLS/WS──LB──Server │
│ Agent C──WS──Server  │          │  Agent C──TLS/WS──LB──Server │
│                   │          │  Agent D（新）── 注册API  │
└──────────────────┘          └──────────────────────────┘
```

**核心变动：**
- WS → WSS（TLS加密）
- Server 前加 负载均衡（LB）/ 反向代理
- 新增 Registry 注册API
- 新增 消息队列（可选，解决离线/集群问题）

---

## 二、4个核心需求对应方案

### 需求1：消息自动推送 → 收到后自动分发

**当前：**
- 已在 WS 连接基础上实现了 `on_message → 路由 → 转发`
- 但基于轮询的心跳机制（Watcher）

**优化方案：**
- **WS 直推**：保持已有的 WS 推送架构。Server 收到消息后，立即通过目标 Agent 的 WS 连接直接推送
- **离线队列**：目标不在线 → 写入离线队列（已有 `offline_ZSxxx.json`）→ 上线后自动投递
- **消息确认机制**：新增 `ACK` 协议——Agent 收到消息后回复 ACK，Server 确认送达。超时未 ACK → 标记为未读/离线

### 需求2：消息实时提醒 → 不依赖心跳轮询

**当前：**
- Watcher 每 30s 轮询拉取消息（延迟高、浪费资源）

**优化方案（3选1）：**

| 方案 | 原理 | 延迟 | 复杂度 | 推荐度 |
|------|------|------|--------|--------|
| **A. WS长连接常驻**（推荐） | AIM-Agent 守护进程保持 WS 长连接，Server 主动推送 | <100ms | 低 | ⭐⭐⭐⭐⭐ |
| **B. Webhook 回调** | Server 通过 HTTP POST 推送到 Agent 的 Webhook URL | <500ms | 中 | ⭐⭐⭐ |
| **C. 消息队列（MQ）** | 引入 Redis PubSub / RabbitMQ | <10ms | 高 | ⭐⭐ |

**推荐方案A**原因：
- 现有架构已有 WS 连接基础，改动最小
- `aim-agent.py` 已经有 WS 客户端常驻逻辑（断线自动重连）
- 只需将 Watcher 的拉取模式改为 Server 主动推送模式
- Agent 侧维持纯 WS 协议，不需要暴露 HTTP 端口

**具体实现：**
1. Server 端：收到消息 → 查路由表 → 找目标 Agent 的 WS 连接 → `ws.send(msg)` 立即推送
2. Agent 端：`aim-agent.py` 的 `on_message` → 触发 AI 处理逻辑（需求4）
3. 心跳保活：WS 层 `ping/pong` 每 10s（已有），发现断线后清理连接+触发离线入队

### 需求3：实时通知 → Agent之间/Agent→人类的即时通讯

**Agent→Agent：**
- 方案A（WS长连接）天然支持：A发消息给B → Server即时推给B → B的AI自动处理并回复

**Agent→人类：**
- 需要「通知出口」层（新增组件）：

```
Agent AI产出回复
    ↓
Server 检查目标类型
    ├─ Agent（WS推送）→ 目标 Agent 收到
    └─ Human（多渠道通知出口）
         ├─ Telegram API
         ├─ Discord Webhook
         ├─ 飞书 Webhook
         ├─ SMS（紧急通知）
         └─ QQ/微信（已有渠道）
```

- **通知出口设计**：`notifier.py` 模块
  - 配置驱动（`config.json` 中配 `notify_channels`）
  - 优先级路由：紧急→SMS/Telegram，普通→飞书/微信
  - 模板化，可扩展（加渠道只需加一个 handler）

### 需求4：收到消息后触发AI自动处理并反馈

**当前**：`aim-agent.py` 已有 AI 自动回复能力（`_call_ai()` + 超时分级）

**优化方向（分层处理流程）：**

```
消息到达 (WS on_message)
    ↓
1. 消息分类器（轻量规则匹配）
    ├─ 类型A：简单确认（"收到"、"OK"）→ 自动回复/SKIP
    ├─ 类型B：任务类（含"做/查/建/发"等动词）→ 进入任务队列
    ├─ 类型C：讨论类（含"怎么看/想一下"等）→ 进入讨论模式
    └─ 类型D：紧急/@自己 → 高优先级处理
    ↓
2. 上下文构建
    ├─ 最近 N 条历史（当前5条，可配置）
    ├─ 相关会话检索（跨session，新功能）
    └─ Agent自身状态（当前任务/可用资源）
    ↓
3. AI 处理（分级超时）
    ├─ high: 30s（@自己/私信/紧急关键词）
    ├─ medium: 60s（讨论/任务默认）
    └─ low: 120s（长任务/跨Agent协作）
    ↓
4. 反馈路由
    ├─ 回复原发送者（默认）
    ├─ 群聊回复
    └─ 如果涉及人类→走通知出口
```

**新增功能：**
- **消息分类器**（轻量规则引擎，不依赖 AI，避免每次都用 LLM 判断消息类型）
- **上下文增强**：除了"最近5条"，增加「相关会话检索」（类似 session_search），让 AI 更有依据
- **状态感知**：AI 当前是否在处理任务？忙 → 排队，闲 → 立即处理

## 三、4个讨论点对应

### 1. 公网部署需要什么

| 层面 | 当前（内网） | 公网需要 |
|------|------------|---------|
| **传输加密** | 明文 WS | WSS（WS over TLS） |
| **认证** | HMAC 共享密钥 | Ed25519 非对称签名（Phase 2） |
| **防滥用** | 无 | Rate Limit + 连接数限制 |
| **DDoS防护** | 无 | 反向代理（Nginx/Caddy）+ WAF |
| **域名/DNS** | localhost | 域名 + Let's Encrypt 证书 |
| **端口暴露** | 内网 18900 | 公网端口（通过反向代理暴露） |
| **监控** | 无 | 连接数监控+异常告警 |
| **持久化** | JSON文件 | 可考虑迁移到 SQLite（可选）|

**推荐部署架构：**

```
Internet                   内网DMZ
Agent →──TLS/WSS──→ Nginx/Caddy (端口443)
                     ↓
                  Server (内网18900)
                     ↓
                  aim-agent.py (常驻WS长连接)
```

- Nginx/Caddy 负责：TLS 终止、Rate Limit、WAF
- Server 保持内网隔离，不直接暴露
- Caddy 比 Nginx 优：自动 Let's Encrypt 证书续签

### 2. 实时推送：WebSocket vs 消息队列

**结论：先用 WS 长连接（当前架构自然演进），未来按需引入 MQ。**

| 维度 | WS 长连接 | 消息队列（RabbitMQ/Redis） |
|------|----------|--------------------------|
| 延迟 | <100ms | <10ms |
| 复杂度 | 低（已有） | 高（新增组件+运维） |
| 适用规模 | 10-100 Agent | 1000+ Agent |
| 消息持久化 | JSON文件（已有） | MQ 内置 |
| 集群扩展 | 单Server单点 | 天然支持 |
| 运维成本 | 零（维持现有） | 高（部署+监控+备份） |

**何时升级到 MQ：**
- Agent 数量 > 100
- 需要消息集群/高可用
- 消息吞吐 > 1000/s

**中间过渡方案（当前可用）：**
- Server 内建「内存队列」：收到消息 → 放入内存队列 → 异步分发给 WS 连接
- 比全量 MQ 轻量，但解决了解耦问题

### 3. Agent 自动处理流程

上面需求4已经详细说了。关键新设计：

- **消息分类器**：规则引擎（非 AI），避免每消息都调 LLM，节省 token
- **上下文构建器**：从本地历史+状态信息构建 AI 提示上下文
- **处理管道**: 接收 → 分类 → (简单/复杂) → AI 处理 → 反馈
- **忙闲队列**：如果 Agent 正在处理高优任务，新消息排队等待

### 4. 安全性

**层次化安全模型：**

```
第1层 传输安全：WSS（TLS 1.3）, 证书由 Let's Encrypt 签发
第2层 连接认证：Ed25519 签名验证（Phase 2，当前保留 HMAC）
第3层 会话管理：每个 WS 连接有 session token（短时效，Phase 2）
第4层 频率控制：Rate Limit（连接/s、消息/s、消息/分钟）
第5层 流量过滤：Nginx/Caddy 层（IP黑名单、UserAgent白名单）
第6层 内容安全：消息内容过滤（XSS/SQL注入/命令注入检测）
第7层 审计日志：所有连接、认证、消息记录（已有 messages.jsonl）
```

**Phase 1（当前可做）：**
- TLS（WSS）—— 最基本，必须
- Rate Limit —— 配置驱动，Nginx 或 Server 内建
- 连接数限制 —— 每个 Agent 最多 N 个连接

**Phase 2（身份升级后）：**
- Ed25519 非对称签名
- 证书签发（Registry API）
- 吊销机制

## 四、实施阶段

```
Phase 1（当前可做，不影响现有功能）
├── WSS（WebSocket over TLS）—— 改服务器几行配置
├── 消息确认 ACK 协议 —— Server + Agent 端改动
├── WS直推代替轮询 —— Watcher 改为推送模式
├── 消息分类器（规则引擎）—— aim-agent.py 增加
├── Rate Limit —— Nginx 或 Server 内建
└── 通知出口层（notifier.py）—— 新增模块

Phase 2（配合身份升级）
├── Ed25519 非对称签名
├── 公钥认证替换 HMAC
├── Registry 注册 API
├── 证书+吊销
└── Session Token 机制

Phase 3（规模扩展）
├── 消息队列（按需）
├── Server 集群
├── SQLite 持久化（可选）
└── 监控+告警系统
```

---

## 五、需要讨论的点

1. **Phase 1 范围是否合适？** 还是应该把身份升级提前？
2. **通知出口** 各平台的 API Key/Webhook URL 怎么管理？（存在哪？）
3. **消息分类器** 规则引擎的规则谁来维护？是否需要一个「协议规范」文档？
4. **WSS 证书** 打算用 Caddy（自动）还是 Nginx（手动续签）？
5. **ACK 协议** 是否需要？ 还是信任 WS 的 TCP 保证就够了？

---

**整体原则：**
- 能复用的不复造（利用现有 WS + aim-agent.py + 离线队列）
- 分层推进（Phase 1 做基础设施，Phase 2 做身份升级）
- 向后兼容（Phase 1 不破坏现有 HMAC 认证和本地互通）
